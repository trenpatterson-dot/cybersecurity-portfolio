# Brute Force Attack Detection Lab

## Overview

A home lab built on VMware Workstation Pro that simulates a complete SSH brute force attack from execution through detection. A Kali Linux attacker uses Hydra v9.6 to launch a dictionary credential attack against an intentionally exposed Ubuntu SSH service. Wireshark captures the attack at the network layer in real time while Wazuh SIEM independently detects and alerts on the activity — automatically, without any rule tuning. The lab exercises the full SOC workflow: attack simulation → network-layer visibility → SIEM alerting → incident reporting, and concludes with a formal incident report and layered remediation recommendations.

**Lab environment:** Isolated VMware host-only network — no external systems were targeted.

### Finding 1 — Out-of-Box SIEM Detection Confirmed

Wazuh Rule ID 5760 fired 10 alerts during the Hydra attack with zero manual rule tuning. The full pipeline — Ubuntu log forwarding → Wazuh ingestion → alert generation — worked end-to-end automatically. This confirms that default Wazuh rule coverage is sufficient to detect SSH brute force campaigns without any custom rule authoring.

### Finding 2 — Zero Credentials Compromised

No successful logins were recorded across all 10 alerts. The attack was entirely unsuccessful, validating the lab as safely contained and confirming that detection was triggered by failed attempts rather than a post-compromise event.

### Finding 3 — Dual-Layer Visibility: SIEM + Network

Wireshark independently corroborated the brute force at the network layer via high-frequency TCP SYN bursts from a single source IP to port 22. This signature is detectable at the perimeter without host-based SIEM telemetry, demonstrating that two independent visibility layers — network and host — both observed the same attack event simultaneously.

### Finding 4 — Automatic MITRE ATT&CK Classification

Wazuh automatically mapped all 10 alerts to T1110 — Brute Force / T1110.001 — Password Guessing (SSH). Correct automatic classification enables structured threat-intelligence reporting without manual analyst translation, and supports integration with detection frameworks and threat-hunting workflows.

### Finding 5 — Detection Without Response Is Incomplete

Wazuh Rule ID 5760 (per-failure, Alert Level 5) and Rule ID 5763 (brute force aggregate threshold, higher level) are architecturally distinct. **Rule 5763 did not trigger during this lab.** Without 5763 firing and driving an automated block via Wazuh active response, an attacker with a larger wordlist would continue indefinitely — generating alerts but facing no enforcement action. Detection alone is not a sufficient control.

### Finding 6 — Intentional Misconfiguration Mirrors Real-World Exposure

OpenSSH Server was deployed with no firewall restrictions, no fail2ban rate-limiting, and no key-based authentication requirement. The attack proceeded completely unimpeded and was caught solely by the SIEM. This is consistent with common real-world SSH exposure patterns and validates the detection scenario as realistic.

## Lessons Learned

**1. Default SIEM rules provide real coverage.**
Wazuh detected an active SSH brute force campaign with zero configuration. For a SOC analyst, this reinforces the value of keeping rule sets current and not assuming detection requires custom engineering — but also the need to understand the thresholds at which default rules fire.

**2. Rule 5760 vs. Rule 5763: detection granularity matters.**
Rule 5760 alerts on each individual SSH failure. Rule 5763 escalates when failures cross a count threshold and is the trigger for automated blocking via Wazuh active response. In this lab, 5763 did not fire. Understanding the difference between alert-only rules and action-triggering rules is critical for SOC triage — an analyst seeing only Level 5 alerts must recognize they are watching an ongoing attack that the SIEM is not yet blocking autonomously.

**3. Network and host-based visibility are complementary, not redundant.**
Wireshark and Wazuh detected the same event independently through different mechanisms. A perimeter control (IDS/IPS or firewall log) watching for SYN bursts to port 22 would catch this attack even if the SIEM was offline. Defense-in-depth applies to visibility layers as much as to controls.

**4. Intentional misconfiguration is a valid and important lab design pattern.**
Deploying SSH with no compensating controls was a deliberate choice. It produced a clean, realistic attack scenario and made the detection story straightforward to follow. Knowing *which* controls were absent — and *which* ones would have broken each phase of the attack — is as instructive as the detection result itself.

**5. Remediation requires closing the full attack surface, not just adding a SIEM.**
The complete remediation stack for this scenario is layered: (a) `fail2ban` to rate-limit and auto-block repeated failures; (b) key-based SSH authentication to eliminate password guessing as an attack vector entirely; (c) `PermitRootLogin no` in `/etc/ssh/sshd_config` to eliminate privileged account targeting; (d) Wazuh active response on Rule 5763 to automate IP blocking at the SIEM layer. Each layer addresses a distinct gap that the others do not cover.
